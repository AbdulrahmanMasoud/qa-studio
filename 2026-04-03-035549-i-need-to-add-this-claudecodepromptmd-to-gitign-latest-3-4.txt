  - No .env.example — new contributors don't know what config is needed

  - No input sanitization — project variables are passed directly to page

  - No rate limiting — could be abused to launch unlimited browser instances
  - No CORS configuration beyond hardcoded localhost

  - No database indexes on testId, projectId columns

     Plan: Error Handling & Resilience

     Context

     QA Studio currently has minimal error handling. Backend routes have no
     try-catch around DB operations, SSE streams crash on client disconnect, the
      batch runner hangs on unhandled rejections, and the frontend has no error
     boundaries, no toast notifications, and silently swallows streaming errors.
      This plan adds a resilience layer across both backend and frontend.

     ---
     Phase 1: Backend

     1.1 Fastify Global Error Handler (apps/server/src/server.ts)

     Add after line 22 (after Fastify instance creation), before CORS:
     - app.setErrorHandler() — logs error, returns JSON { error } with
     appropriate status code
     - app.setNotFoundHandler() — returns 404 JSON for unmatched API routes

     This single change protects all unguarded DB operations across every route
     file — they'll return 500 JSON instead of crashing.

     1.2 Fix Batch Runner Unhandled Rejection 
     (apps/server/src/services/batch-runner.ts)

     Line 47: runTest(...).then(...) has no .catch(). If runTest throws in
     parallel mode, the batch hangs forever.
     - Add .catch() that creates a synthetic failed TestRun and continues the
     batch
     - Matches the sequential path behavior (which catches internally via
     runner.ts)

     1.3 Safe SSE Writes (apps/server/src/routes/tests.ts, 
     apps/server/src/routes/suites.ts)

     Create helper in apps/server/src/utils/sse.ts:
     function safeSSEWrite(raw: ServerResponse, data: object): boolean
     - Checks raw.destroyed || raw.writableEnded before writing
     - Wraps raw.write() in try-catch
     - Replace all reply.raw.write(...) calls in tests.ts (2 calls) and
     suites.ts (6 calls)
     - Also wrap the final reply.raw.end() calls

     1.4 Visual Diff Error Handling (apps/server/src/services/visual-diff.ts)

     Wrap compareScreenshots body in try-catch:
     - readFileSync can throw if file missing/corrupt
     - PNG.sync.read can throw on invalid PNG data
     - Re-throw with contextual message: "Visual diff failed: 
     ${originalMessage}"

     1.5 Migration Logging (apps/server/src/db/index.ts)

     Change empty catch {} blocks on ALTER TABLE statements to log unexpected
     errors (ignore "duplicate column" which is expected).

     ---
     Phase 2: Frontend Foundation

     2.1 Install Toast Library

     Add sonner to apps/dashboard — lightweight (~3KB), Tailwind-native,
     supports success/error/loading states.

     2.2 Toast Provider + Global Mutation Error Handler 
     (apps/dashboard/src/main.tsx)

     - Add <Toaster position="bottom-right" richColors /> to the render tree
     - Add mutations.onError to QueryClient defaultOptions → auto-shows toast
     for every failed mutation
     - Add queries.retry: 1 for a single query retry

     2.3 React Error Boundary (new: 
     apps/dashboard/src/components/ErrorBoundary.tsx)

     - Class component with getDerivedStateFromError + componentDidCatch
     - Fallback UI with error message and "Go to Projects" reset button
     - Uses existing Tailwind error patterns (bg-red-100, AlertCircle icon)

     2.4 404 Route + Mount Error Boundary (apps/dashboard/src/App.tsx)

     - Add <ErrorBoundary> wrapping <Routes>
     - Add <Route path="*"> catch-all with a simple 404 page inside Layout
     - Inline NotFound component (no separate file needed)

     ---
     Phase 3: Frontend Error Fixes

     3.1 Success Toasts on Mutations

     Add toast.success() in onSuccess for key mutations across:
     - apps/dashboard/src/pages/ProjectsPage.tsx — create, delete
     - apps/dashboard/src/pages/TestsPage.tsx — create, clone, delete, env vars
     save

⏺ Here's how Visual Baselines work in QA Studio:

  How It Works

  1. Capture Screenshots

  Add Screenshot steps to your test. When the test runs, Playwright captures a
  PNG at each screenshot step and stores it in data/screenshots/.

  2. Set Baselines

  After a passing run that has screenshot steps, open the Baselines panel (the
  eye icon in the test builder). You'll see a list of passing runs — click Set 
  Baselines from Run to save those screenshots as the "golden" reference images.

  Under the hood (visual-regression.ts:17-57):
  - Deletes any existing baselines and their associated diffs for this test
  - Creates a new baselines record per screenshot step, pointing to that run's
  screenshot file

  3. Automatic Comparison on Future Runs

  Every time you run the test again, the runner (runner.ts) automatically
  compares each screenshot step's output against its baseline using pixelmatch:

  - Reads both PNGs (baseline + actual)
  - If dimensions differ, pads the smaller image with white pixels
  - Runs pixel-level comparison (threshold: 0.1 sensitivity)
  - Calculates a diff percentage (mismatched pixels / total pixels)
  - Generates a red/orange diff image highlighting changes
  - Stores the result in screenshot_diffs with status: match (≤5%) or mismatch
  (>5%)

  4. Review Diffs

  In the run detail panel, you can see visual diffs for any run. Each diff
  shows:
  - Baseline image (what it should look like)
  - Actual image (what this run captured)
  - Diff image (red/orange overlay showing changes)
  - Diff percentage (e.g., 2.3%)

  5. Approve or Reject

  - Approve — accepts the visual change as intentional. The baseline is updated
  to the new screenshot so future runs compare against it.
  - Reject — marks the change as a regression. The baseline stays the same.

  Key Details

  ┌───────────────────────┬────────────────────────────────┐
  │        Setting        │             Value              │
  ├───────────────────────┼────────────────────────────────┤
  │ Comparison library    │ pixelmatch                     │
  ├───────────────────────┼────────────────────────────────┤
  │ Match threshold       │ ≤5% diff = match               │
  ├───────────────────────┼────────────────────────────────┤
  │ Pixel sensitivity     │ 0.1 (pixelmatch threshold)     │
  ├───────────────────────┼────────────────────────────────┤
  │ Dimension handling    │ Pads smaller image with white  │
  ├───────────────────────┼────────────────────────────────┤
  │ Diff highlight colors │ Red + orange                   │
  ├───────────────────────┼────────────────────────────────┤
  │ Storage               │ data/screenshots/, data/diffs/ │
  └───────────────────────┴────────────────────────────────┘

  1. Add Screenshot steps to test
  2. Run test → passes
  3. Open Baselines → "Set from Run"
  4. Make UI changes to your app
  5. Run test again → diffs generated automatically
  6. Review diffs → Approve (update baseline) or Reject (flag regression)
